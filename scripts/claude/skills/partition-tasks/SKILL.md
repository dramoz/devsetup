## When This Skill Applies

- User says `/partition_tasks` or `/partition-tasks`
- User asks "how many phases fit per session / per run / per window"
- User wants to split an implementation plan into batched `claude -p` runs
- User wants a shell script that chains `claude -p "..."` with `sleep` gaps
- User asks about Pro quota limits and how to work within them

## Arguments

`$ARGUMENTS` may specify:
- A path to the plan file (default: auto-detect from `plans/`)
- A model: `sonnet` (default) or `opus`
- A window size in minutes (default: `300` = 5 h)
- An output script name (default: `run_partitioned.sh`)

Example invocations:
```
/partition_tasks
/partition_tasks plans/AUDIT_IMPLEMENTATION_PLAN.md
/partition_tasks plans/PLAN.md opus 300
```

## Execution Steps

### Step 1 — Locate the plan file

If `$ARGUMENTS` includes a file path, use it. Otherwise glob for the most recently
modified file matching `plans/*PLAN*.md` or `plans/*plan*.md` in the current
working directory. Read the file completely.

### Step 2 — Identify phases and tasks

Parse the plan for phase headings and task rows. For each phase extract:
- Phase number / name
- Number of tasks
- Whether tasks are `[P]` (parallel) or `[S]` (sequential)
- Rough "weight" of each task: number of files it touches, whether it writes new
  files vs edits existing ones, whether it runs tests

### Step 3 — Estimate tool-call budget per model

Use these empirical estimates for Claude Pro 5-hour windows:

| Model   | Approx. tool calls / 5 h window |
|---------|----------------------------------|
| sonnet  | 150–200                          |
| opus    | 50–70                            |

**Per-task cost heuristic** (tool calls):

| Task type                              | Cost |
|----------------------------------------|------|
| Read a file                            | 1    |
| Edit an existing file (small patch)    | 2    |
| Edit an existing file (large section)  | 3    |
| Write a new file                       | 2    |
| Run tests / bash command               | 1    |
| Git commit                             | 1    |
| Per additional module touched          | +1   |

Apply a **1.4× overhead multiplier** to account for context reads, re-reads,
plan file updates, and error recovery.

### Step 4 — Group phases into batches

Assign phases greedily to batches, never exceeding 80% of the model's window
budget per batch (leave 20% headroom for error recovery):

- sonnet: target ≤ 140 tool calls per batch
- opus:   target ≤ 45 tool calls per batch

Keep phases that depend on each other (sequential in the plan) in the same batch
where possible. Never split a single phase across batches.

### Step 5 — Write the shell script

Write a script to `$OUTPUT` (default `run_partitioned.sh`) with this structure:

```bash
#!/bin/bash
set -euo pipefail

# Auto-generated by /partition_tasks
# Plan : <plan file>
# Model: <model>
# Date : <today>

COMMON_FLAGS="--model <model> --dangerously-skip-permissions"
PROJECT_DIR="$(pwd)"
SLEEP_BETWEEN="<window_minutes>m"

echo "=== Run 1/N: Phases X-Y (<label>) ==="
cd "$PROJECT_DIR"
claude -p "<prompt for batch 1>" $COMMON_FLAGS

echo "=== Run 1 complete. Sleeping ${SLEEP_BETWEEN}... ==="
sleep "$SLEEP_BETWEEN"

echo "=== Run 2/N: Phases ... ==="
...

echo "=== All N runs complete. ==="
```

**Prompt construction for each batch:**

Each `claude -p "..."` prompt must include:
1. "Execute phases X, Y, Z from <plan file>."
2. One paragraph per phase describing the tasks and their `[S]`/`[P]` ordering.
3. The standard tail instruction:
   > "For each task: implement, run `QT_QPA_PLATFORM=offscreen python -m pytest --tb=short -q`
   > to verify no regressions, commit with a descriptive message. After each phase run the
   > verification checkpoint from the plan. Mark completed tasks with checkmarks in the plan
   > file and commit the plan update."
   (Adapt the test command if the project's CLAUDE.md specifies a different test runner.)

### Step 6 — Make the script executable and report

```bash
chmod +x <output script>
```

Then report to the user:
- Number of batches created
- Phases per batch with estimated tool-call cost
- Total estimated wall-clock time (batches × window + sleeps)
- The output script path
- Any phases that were too large to fit in one window (flag for manual splitting)

## Notes

- Always prefer **sonnet** unless the user explicitly requests opus. Sonnet has ~3×
  the quota headroom on Pro.
- If a single phase exceeds the per-batch budget on its own, flag it and suggest
  splitting it into sub-tasks in the plan before running.
- The `sleep` duration equals the quota window (default 5 h = 300 m). If the user
  has already partially consumed their window, they should adjust manually.
- Do NOT push to remote — only local commits.
- The generated script uses `nohup`-safe patterns (`set -euo pipefail`, no
  interactive input required).
